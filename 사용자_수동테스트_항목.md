# 🧪 사용자 수동 테스트 필요 항목

> Claude가 자동 검증할 수 없는 시나리오. 실제 Instagram/Stripe/Kakao 연동이나 PRO 플랜, 실제 이메일 SMTP 등이 필요.

## 🔴 필수 수동 테스트 (배포 전)

### 1. Instagram DM 수신 → 자동응답 End-to-End
**왜 필요한가**: 센드잇의 핵심 기능. 실제 Instagram Business API에서 webhook 이벤트를 받아 키워드 매칭 → 자동응답 발송까지 전체 흐름.

**테스트 시나리오**:
- [ ] **S19 키워드 매칭 실동작 확인**
  - `가격` keyword (CONTAINS) 트리거 생성
  - 실제 IG 계정으로 테스트 계정에 DM "가격 알려주세요" 전송
  - 자동응답 수신 여부 확인
  - 같은 키워드로 EXACT/STARTS_WITH 매칭도 동작하는지 (S19 fix 후 STARTS_WITH 포함)
- [ ] **S20 댓글 트리거 postId 지정 vs 전체**
  - postId 빈 값 → 모든 게시물 댓글 발동
  - postId 지정 → 해당 게시물만 발동
  - 잘못된 postId 저장 시도 → 검증 안내 여부
- [ ] **S23 비활성 트리거 발동 차단**
  - 활성 트리거로 매칭 성공 확인
  - 비활성 토글 후 동일 키워드 DM → 응답 오지 않아야 함
- [ ] **S33 라이브챗 메시지 전송 & 자동→수동 전환**
  - 실제 DM 수신 대화 클릭 → 수동 메시지 입력 → 전송
  - "자동화 일시정지" 토글 → 이후 같은 키워드 DM이 자동응답 안 받는지
  - 이미지/카드/빠른답변 버튼 실제 발송

**선결 조건**:
- Meta 앱 승인 완료 (`instagram_manage_messages` 권한)
- 테스트용 Instagram Business Account + Facebook Page
- Webhook subscription 설정
- 공인 IP/ngrok 등 콜백 수신 가능 환경

### 2. 결제 & 플랜 업그레이드 (Stripe)
**왜 필요한가**: PRO/ENTERPRISE 전용 기능 대부분이 이 플로우에 의존.

**테스트 시나리오**:
- [ ] **S46 Stripe Checkout 플로우**
  - 설정 → 결제 → "프로 업그레이드" 클릭
  - Stripe 결제 화면 진입 (또는 Mock)
  - 결제 완료 후 polling 2초×3회 → `plan: PRO` 갱신 확인
  - 결제 실패/취소 시 안내
- [ ] **S48 PRO 플랜 진입 후 확인**
  - 브로드캐스트 페이지 모달 사라짐
  - 시퀀스 빌더 접근 가능
  - 에이전시 "계정 추가" (한도 확인)

**선결 조건**:
- Stripe 테스트 키 (`STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`) 환경변수 설정
- 백엔드 `BillingController` Stripe Webhook 라우트 설정
- 테스트 카드 (`4242 4242 4242 4242`)

### 3. 이메일 인증 (SMTP)
**왜 필요한가**: 기존 1차 리포트에서 SMTP 미설정 시 가입 진행 막힘 — 미인증 로그인 우회 버그로 "진행은 되지만" 정상 플로우 아님.

**테스트 시나리오**:
- [ ] **S2 회원가입 인증 코드 수신**
  - 실 이메일 주소로 가입
  - 메일함에서 6자리 인증 코드 수신 확인
  - 코드 만료 후 재발송 동작
  - 재발송 남용 방지 (rate limit)
- [ ] **S18 비밀번호 재설정 메일**
  - 로그인 화면 "비밀번호를 잊으셨나요"
  - 이메일 입력 → 재설정 메일 수신
  - 링크/코드로 새 비밀번호 설정
  - 같은 코드 재사용 차단

**선결 조건**:
- `application.yml`의 SMTP 설정 (Gmail/Naver/SES 등)
- `EmailService` `sendVerificationEmail` 정상 동작

## 📝 수동 테스트 방법 공통 팁

1. **테스트 계정 준비**:
   - Instagram Business: `@sendit_test_kr` 같은 별도 계정
   - Facebook Page: 개인 페이지가 아닌 비즈니스 페이지
   - 이메일: `+` alias 활용 (`myemail+qa1@domain.com`)

2. **이슈 발견 시 기록 양식** (기존 시나리오.md 상단 "발견 이슈 기록 양식" 참고):
   ```
   ## [시나리오번호] 이슈명
   - 심각도: Critical / High / Medium / Low
   - 재현 절차: ...
   - 현재 동작: ...
   - 기대 동작: ...
   - 관련 파일: ...
   - 비고: ...
   ```
   → `시나리오.md` 하단 "3차 검증 이슈 리포트" 섹션 신설 권장.

3. **H2 리셋 주의**: 백엔드 재시작 시 모든 데이터 소실. `application.yml`에서 `ddl-auto: update` + 파일 기반 H2 또는 PostgreSQL로 전환 시 지속성 확보.

4. **Webhook 개발환경**:
   - ngrok으로 로컬 백엔드 노출
   - Meta 앱 대시보드에서 webhook URL 등록
   - `verify_token` 일치 확인
