## Phase 1: 데이터 스키마 확장 + 프론트 저장/로드 수정 (먼저 이거부터!)

### 목표
그래프 정보가 백엔드까지 온전하게 전달되고 되돌아오게 한다.
Phase 1만 끝나도 "저장해도 엣지가 사라지는 버그"는 해결됨.

### 구현 순서

1. **flowSerializer.js v2 스키마 도입**
   - `graphToFlowData(nodes, edges)`를 아래 JSON 구조로 재작성:
     ```json
     {
       "version": 2,
       "nodes": [
         { "id": "trigger-1", "type": "trigger", "data": { ... }, "position": {...} },
         ...
       ],
       "edges": [
         { "id": "e1", "source": "trigger-1", "target": "msg-1", "sourceHandle": null },
         { "id": "e2", "source": "cond-1", "target": "act-1", "sourceHandle": "pass" },
         ...
       ]
     }
     ```
   - data는 프론트 노드의 data를 그대로 전달 (role, conditionType, retryOnFail, retryButton, specificPostUrl 등 모두 포함)
   - position도 보존 (사용자가 저장 후 다시 열었을 때 레이아웃 유지)

2. **flowDataToGraph() v2 우선 처리 + v1 fallback**
   - if (data.version === 2) { return { nodes: data.nodes, edges: data.edges } }
   - else { 기존 v1 변환 로직 (requirements/mainDm/followUp/... → nodes/edges 재구성) }
   - 테스트용 기존 v1 플로우 하나 만들어 로드되는지 확인

3. **저장 전 그래프 검증**
   - trigger 노드가 반드시 1개 존재
   - 모든 edge의 source/target이 nodes 배열의 id 중에 있음
   - 순환이 있으면 경고 (무한 루프 방지)
   - 검증 실패 시 toast로 사용자에게 안내

4. **백엔드 Flow.flowData 컬럼**
   - 기존 그대로 JSON 문자열 (스키마 변경 없음, 내용만 v2)
   - DB 마이그레이션 불필요

5. **Preview 검증**
   - 분기 플로우(예: 팔로우 확인 → pass/fail 두 경로) 만들어 저장 → 재로드
   - 엣지와 sourceHandle이 정확히 복원되는지 확인
   - 기존 저장된 v1 플로우 있으면 자동 변환 확인

### Phase 1 완료 기준
- [ ] graphToFlowData가 nodes + edges + position + version 모두 보존
- [ ] flowDataToGraph가 v2/v1 둘 다 정확히 처리
- [ ] 저장 → 재로드 시 캔버스 완전 동일 (position 포함)
- [ ] trigger 누락/끊긴 edge 등 유효성 검증 토스트 동작
- [ ] Preview에서 분기 플로우 저장/로드 roundtrip 성공
- [ ] 단위 테스트(선택): flowSerializer.test.js에 roundtrip 케이스 2~3개

### Phase 1 주의사항
- 백엔드 FlowExecutionService는 이번 Phase에선 건드리지 않음.
  여전히 v1 구조를 기대하므로, 프론트가 저장하는 flowData를 백엔드가 못 읽어도 괜찮다.
  (다음 Phase 2에서 백엔드 엔진을 재작성할 것이므로)
- 단, 기존 v1을 저장하던 "온보딩 마법사" / "템플릿"이 깨지면 안 됨 —
  이들은 직접 백엔드 API를 호출하거나 별도 시드가 있는지 확인 후 영향 없도록.
