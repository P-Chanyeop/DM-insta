# 🏗️ 플로우 빌더 대대적 리팩터링 계획

> **결정**: 프론트의 그래프(nodes + edges)를 **있는 그대로** 백엔드에 저장하고, 백엔드도 그래프를 순회 실행하는 구조로 재작성한다.
>
> **배경**: 2026-04-17 감사 결과, UI에 존재하는 노드 중 ~30%가 백엔드 no-op이고, 엣지/sourceHandle이 저장조차 안 되어 분기 플로우가 전혀 동작하지 않음. 런칭 전 반드시 해결 필요.

---

## 🎯 목표

1. **모든 UI 노드가 백엔드에서 실제 실행**되어야 한다 (no-op 제거)
2. **엣지와 sourceHandle이 보존**되어 pass/fail, a/b 분기가 실제로 작동해야 한다
3. **한 플로우에 같은 타입 노드 여러 개**를 둘 수 있어야 한다 (싱글톤 제거)
4. **기존 저장된 플로우와 호환** 또는 명시적 마이그레이션 경로 제공

## 🏛️ 새 아키텍처 — 그래프 기반 실행

### 데이터 모델 변경

**Before (고정 슬롯)**:
```json
{
  "trigger": {...},
  "requirements": { "followCheck": {...}, "emailCollection": {...} },
  "mainDm": {...},
  "followUp": { "delay": 30, "unit": "minutes", "message": "..." },
  "conditions": [...],
  "abtest": {...},
  "inventory": {...},
  "carousel": {...}
}
```

**After (그래프)**:
```json
{
  "version": 2,
  "nodes": [
    { "id": "trigger-1", "type": "trigger", "data": {...} },
    { "id": "msg-1", "type": "message", "data": { "role": "opening", ... } },
    { "id": "cond-1", "type": "condition", "data": { "conditionType": "followCheck", ... } },
    { "id": "act-1", "type": "action", "data": { "actionType": "addTag", "value": "VIP" } },
    ...
  ],
  "edges": [
    { "id": "e1", "source": "trigger-1", "target": "msg-1" },
    { "id": "e2", "source": "msg-1", "target": "cond-1" },
    { "id": "e3", "source": "cond-1", "target": "act-1", "sourceHandle": "pass" },
    { "id": "e4", "source": "cond-1", "target": "end-1", "sourceHandle": "fail" }
  ]
}
```

### 백엔드 실행 엔진 설계

```java
// 핵심 인터페이스
public interface NodeExecutor {
    NodeExecResult execute(FlowContext ctx, FlowNode node);
}

// 각 노드 타입별 구현
@Component public class MessageNodeExecutor implements NodeExecutor { ... }
@Component public class ConditionNodeExecutor implements NodeExecutor { ... }
@Component public class ActionNodeExecutor implements NodeExecutor { ... }
@Component public class WebhookNodeExecutor implements NodeExecutor { ... }
@Component public class DelayNodeExecutor implements NodeExecutor { ... }
@Component public class CarouselNodeExecutor implements NodeExecutor { ... }
@Component public class AbTestNodeExecutor implements NodeExecutor { ... }
@Component public class AiResponseNodeExecutor implements NodeExecutor { ... }
@Component public class KakaoNodeExecutor implements NodeExecutor { ... }
@Component public class OptInNodeExecutor implements NodeExecutor { ... }
@Component public class InventoryNodeExecutor implements NodeExecutor { ... }

// 결과 — 다음 노드 선택에 사용
public class NodeExecResult {
    public boolean success;
    public String branch;  // "pass"/"fail"/"a"/"b"/"default"
    public boolean haltFlow;  // 여기서 플로우 종료
    public boolean awaitUser;  // postback/message 대기 (후속은 WebhookEventService가 resume)
    public String resumeToken;  // 사용자 응답 매칭용
}

// FlowExecutionService 재작성
public void executeFlow(Flow flow, FlowContext ctx) {
    Map<String, FlowNode> byId = ...;
    Map<String, List<FlowEdge>> outgoing = ...;  // sourceId -> edges
    
    String currentId = findTriggerNodeId(flow);
    int steps = 0;
    while (currentId != null && steps++ < MAX_STEPS) {
        FlowNode node = byId.get(currentId);
        NodeExecutor executor = registry.get(node.type);
        NodeExecResult result = executor.execute(ctx, node);
        
        if (result.haltFlow) break;
        if (result.awaitUser) {
            pendingActionRepository.save(...);
            break;
        }
        
        // 다음 노드 선택 — branch handle 매칭
        List<FlowEdge> outs = outgoing.get(currentId);
        currentId = chooseNext(outs, result.branch);
    }
}
```

### 핵심 이점
- 각 노드 실행이 **독립 Executor**로 분리 → 테스트/확장 용이
- **엣지와 sourceHandle로 분기** → pass/fail, a/b, success/error 전부 지원
- **노드 순서는 그래프 구조에서 나옴** → 싱글톤 불필요
- **awaitUser** 플래그로 "팔로우 확인" 같은 대기 노드 자연스럽게 처리

## 📅 단계별 작업 계획 (총 5 Phase)

### Phase 1: **데이터 스키마 확장 + 프론트 저장/로드 수정** (4~6h)

**목표**: 그래프 정보가 백엔드까지 온전하게 전달되고 되돌아오기.

**작업**:
- `flowSerializer.js` `graphToFlowData()` 완전 재작성
  - nodes 배열 그대로 저장 (id, type, data 모두)
  - edges 배열 그대로 저장 (id, source, target, sourceHandle)
  - `version: 2` 필드 추가
- `flowDataToGraph()` v2 우선 처리, v1 레거시 fallback
- 백엔드 `Flow.flowData` JSON 필드는 그대로 (스키마 변경 없음, 내용만 다름)
- 저장 전 검증: trigger 존재, 모든 edge의 source/target이 노드 목록에 있음

**검증**:
- 캔버스 그린 그래프 → 저장 → 재로드 시 노드/엣지 완전 동일
- 기존 v1 플로우 로드 시 자동으로 v2로 마이그레이션

### Phase 2: **백엔드 실행 엔진 재설계** (6~8h)

**작업**:
- `FlowNode`, `FlowEdge`, `FlowGraph` DTO 신설 (`backend/.../dto/flow/`)
- `NodeExecutor` 인터페이스 + `NodeExecutorRegistry` 컴포넌트
- `FlowExecutionService` 재작성
  - 기존 `sendMainDmAndFollowUp()` 등 삭제
  - 새 `executeFlowGraph(flow, ctx)` 그래프 순회 메서드
  - `MAX_STEPS = 50`, 순환 방지 visited set
  - postback/message 대기 시 `PendingFlowStep` 엔티티로 중단점 저장
- 기존 테스트(`FlowExecutionServiceTest`) 복구 + 신규 테스트 추가

**검증**:
- 단위 테스트: 선형 플로우, 분기 플로우(pass/fail), 다중 액션, delay, webhook 실행
- 통합 테스트 시나리오: 트리거 → 팔로우 확인 → 태그 추가 → 메인 DM → 대기 → 팔로업

### Phase 3: **각 노드 Executor 구현 — Critical부터** (10~14h)

**순서**:
1. **MessageNodeExecutor** (opening/main/followup) — 기존 로직 이식
2. **ConditionNodeExecutor** (followCheck/emailCheck/tagCheck/customField/timeRange/random) — 평가 후 branch 결정
3. **ActionNodeExecutor** ⚡ NEW — addTag/removeTag/setVariable/addNote/subscribe/unsubscribe
4. **WebhookNodeExecutor** ⚡ NEW — RestTemplate으로 실제 HTTP 호출
5. **DelayNodeExecutor** ⚡ NEW — 독립 delay 노드 지원 (ScheduledFollowUp 활용)
6. **CarouselNodeExecutor** — 기존 로직 이식
7. **AbTestNodeExecutor** — variant별 branch 결정 + message 선택
8. **AiResponseNodeExecutor** — 기존 AIService 위임
9. **KakaoNodeExecutor** — 기존 KakaoChannelService 위임
10. **OptInNodeExecutor** — 기존 RecurringNotificationService 위임
11. **InventoryNodeExecutor** — hasStock 분기 (✅ pass / ❌ fail handle)

**각 executor마다**:
- 단위 테스트 (mock으로 API 호출 검증)
- 성공/실패/awaitUser 3가지 결과 경로 커버

### Phase 4: **프론트 연동 보강** (4~6h)

**작업**:
- `NodeEditor` 필수 필드 검증 (webhook url, kakao templateCode, condition 필수 필드 등)
- `trigger.specificPostUrl` 저장 경로 복구
- `condition.followCheck.retryOnFail` + `retryButton` 직렬화
- `abtest` variant별 메시지 필드 추가 + variantA/variantAPercent 통일
- `inventory` 재고 유무 pass/fail handle 추가 (UI + serializer)
- MiniMap 색상 맵에 inventory/optIn/kakao 추가
- 플로우 저장 시 "미연결 경고" 배너 (trigger가 leaf 노드로 이어지지 않으면 경고)

### Phase 5: **마이그레이션 + 정리** (3~4h)

**작업**:
- 기존 DB에 저장된 v1 플로우 로드 시 자동 v2 변환 로직 검증
- `FlowExecutionServiceTest` 전체 복구 + 카테고리별 시나리오 테스트 추가

## 🚨 위험 요소 및 대응

| 위험 | 대응 |
|---|---|
| 기존 저장 플로우 호환성 | v1 → v2 자동 변환 로직 + 변환 실패 시 관리자 알림 |
| 그래프 순환으로 무한 루프 | `MAX_STEPS=50` + visited set + self-loop 허용은 followCheck retry에만 |
| 테스트 부재 | 각 executor별 단위 테스트 필수 작성 |
| 기존 webhook 이벤트 처리 중단 | `WebhookEventService.handlePostback()` 등의 경로는 유지, 내부에서 `executeFlowGraph` 호출만 교체 |
| Instagram API rate limit | 각 executor가 자체 retry/backoff, WebhookEventService의 중복 실행 캐시 유지 |
