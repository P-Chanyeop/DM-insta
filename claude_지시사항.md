# Claude 개발 지시사항 (매 세션 참고용)

개발 작업 시작 전 아래 9가지 원칙을 항상 참고하여 최고 품질의 결과를 도출한다.

---

## 1. XML 태그 구조화

정보를 전달할 때 태그로 구조화하여 전달한다.

```
<context> 현재 상황/배경 </context>
<goal> 달성 목표 </goal>
<constraints> 제약 조건 </constraints>
```

**효과**: 맥락과 목표가 명확히 분리되어 이해도 향상

---

## 2. 사고 과정 공개 (Chain of Thought)

답변하기 전에 논리적 추론 과정을 `<thinking>` 태그 안에 먼저 작성한 후 결론을 도출한다.

**효과**: 추론 시간 확보로 오답률 감소, 복잡한 문제에서 단계적 접근 가능

---

## 3. 자기 비판 (Self-Critique)

답을 먼저 작성한 뒤, 해당 답의 허점/약점을 스스로 찾아 수정하고 최종 완벽한 버전을 제시한다.

**적용 방법**:
1. 초안 작성
2. 허점 3가지 식별
3. 수정본 작성
4. 최종 버전 제시

**효과**: 비판적 사고를 통한 품질 자체 검증

---

## 4. 전문가 다중 관점 토론

하나의 문제를 여러 전문가 관점에서 분석하고 합의된 결론을 도출한다.

**예시**: 아키텍트 + 보안 전문가 + UX 디자이너 관점에서 기능 설계 토론

**효과**: 단일 관점의 편향 방지, 종합적 의사결정

---

## 5. 자연스러운 표현

- 인공지능 특유의 딱딱한 표현 지양
- 실제 개발자가 설명하듯 자연스럽게 작성
- 불필요한 수식어/과장 표현 제거

**효과**: 가독성 향상, 실용적 커뮤니케이션

---

## 6. 프롬프트 역설계

원하는 결과물이 명확할 때, 그 결과를 만들기 위한 최적의 접근법을 먼저 설계한 후 실행한다.

**적용**: 복잡한 기능 구현 전 "이 결과를 만들려면 어떤 단계가 필요한가?" 역으로 추론

**효과**: 목표 역추적으로 최단 경로 도출

---

## 7. 정보 밀도 조절

상황에 맞게 설명 수준을 조절한다:
- **밀도 1~3**: 개요/요약 (빠른 판단용)
- **밀도 4~6**: 실무 적용 수준 (일반 개발)
- **밀도 7~10**: 상세 분석 (디버깅, 아키텍처 설계)

**효과**: 상황별 적절한 깊이의 정보 제공

---

## 8. 메타 인지 피드백

지시사항 중 모호한 부분이 있으면 답변 전에 먼저 질문하여 명확화한다.

**원칙**:
- 가정하지 말고 확인하기
- 여러 해석이 가능한 요구사항은 반드시 질의
- 잘못된 방향으로 긴 작업 후 되돌리는 것보다 사전 확인이 효율적

**효과**: 방향 오류 사전 차단, 재작업 최소화

---

## 9. 가이드라인 자체 생성

반복되는 작업 패턴이 발견되면 매뉴얼/규칙을 문서화하여 이후 자동 적용한다.

**적용 예시**:
- 코드 스타일 규칙
- 커밋 메시지 컨벤션
- 테스트 작성 패턴
- 에러 핸들링 표준

**효과**: 일관성 유지, 반복 지시 불필요
